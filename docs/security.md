# Безопасность

Аутентификация по JWT, логин — email с обязательным подтверждением, защита от перебора,
ролевая авторизация и ограничения видимости для админов.

## Аутентификация

- Логин — **email**, он же `UserName` учётной записи; уникален (`User.RequireUniqueEmail`).
- Вход выдаёт **пару токенов**:
  - **access JWT** (HS512, claim-ы id+роли), короткоживущий — `Jwt:AccessTokenMinutes` (по умолчанию 120 мин);
  - **refresh-токен** — долгоживущий (`Jwt:RefreshTokenDays`, 180 дней), **хранится в БД** (таблица
    `RefreshTokens`, только SHA-256 хэш) и **отзываем**.
- `POST /api/auth/refresh` меняет refresh на новую пару (**ротация**: старый отзывается). Повторное
  предъявление уже отозванного токена трактуется как кража и отзывает все токены пользователя.
- `POST /api/auth/logout` отзывает refresh.
- **Отзыв доступа**: refresh-токены отзываются при смене/сбросе пароля (`RevokeAllAsync`) и удаляются
  каскадом при удалении учётной записи (удаление админа). После этого новый access не выдаётся, а
  текущий перестаёт действовать максимум через `AccessTokenMinutes`. Реализация — `IRefreshTokenService`
  / `RefreshTokenService`; оркестрация в `AuthService`.
- Фронт хранит обе части (`TokenStore`), **проактивно обновляет** access по истечении (`AuthHeaderHandler`
  + `TokenRefresher`, отдельный клиент без хендлера — без рекурсии), при выходе дёргает `logout`.
- Защищённые ручки требуют заголовок `Authorization: Bearer <token>`. В Scalar токен вводится
  кнопкой авторизации (схема Bearer добавляется в OpenAPI document-трансформером).

## Подтверждение почты

Вход **запрещён без подтверждённой почты** (`SignIn.RequireConfirmedEmail`).

```
register → на почту уходит 6-значный код → confirm-email → можно login
```

- Код — TOTP через `EmailTokenProvider` (purpose `EmailConfirmation`), живёт ~10 минут,
  без хранения в БД.
- `resend-confirmation` отправляет код повторно и **не раскрывает**, существует ли такой email.

## Сброс пароля

```
forgot-password → на почту уходит 6-значный код → reset-password(code, newPassword)
```

- Код — TOTP (purpose `ResetPassword`). `forgot-password` не раскрывает существование email.
- Успешный сброс дополнительно подтверждает почту (владение ящиком доказано) и снимает счётчик
  блокировки. Смена пароля обновляет security stamp — ранее выданные коды инвалидируются.

## Неподтверждённые аккаунты

- Регистрация создаёт студента в статусе `IsActive=false`; он не виден в рейтинге/очередях/поиске,
  пока почта не подтверждена.
- Повторная регистрация на тот же **неподтверждённый** email удаляет прежнюю запись и создаёт новую
  (чтобы пользователь не «застрял»); на **подтверждённый** email — `409`.
- Фоновый сервис (`UnconfirmedAccountCleanupService`) каждые 10 минут удаляет неподтверждённые аккаунты
  старше 30 минут (вместе с профилем — каскадом по учётной записи).
- Лимит: не чаще **одного письма в минуту на email** (`IEmailThrottle`); превышение → `429`.

## Защита от перебора

- **Identity lockout**: 5 неудачных попыток входа → блокировка на 15 минут
  (`Lockout.MaxFailedAccessAttempts` / `DefaultLockoutTimeSpan`). Реализовано в
  `IdentityService.ValidateCredentialsAsync` (`AccessFailedAsync` / `IsLockedOutAsync`).
- **Rate limiting**: все `/api/auth/*` под политикой `auth` — фиксированное окно 10 запросов/мин
  с одного IP, превышение → `429`. См. `RateLimitPolicies.Auth` и настройку в `Program.cs`.
- Ответы логина не различают «нет пользователя» и «неверный пароль» (`InvalidCredentials`),
  отдельно сигнализируются `EmailNotConfirmed` и `LockedOut`.

## Роли и политики

Роли (`Roles`): `User` (студент), `Admin`, `SuperAdmin`. Политика авторизации
`AuthorizationPolicies.SuperAdmin` (`"RequireSuperAdminRole"`) требует роль `SuperAdmin`.

| Уровень | Кто | Примеры прав |
|---------|-----|--------------|
| Аноним | 🌐 | Списки групп/предметов/задач, публичная карточка студента, сводная таблица результатов |
| Студент (`User`) | 👤 | Свой профиль, свои предметы/группа/результаты, запись в очередь |
| Админ (`Admin`) | 🛡 | Создание предметов/задач/групп/событий, проверка сдач, оценки, контакты студентов, очереди |
| Супер-админ (`SuperAdmin`) | ⭐ | Управление админами, глобальные ограничения, удаление предметов/задач/групп, бэкап БД |

Точечная защита в контроллерах: на `join` в очередь и на `me/profile` админ получает `403`
(действия для студента), а удаление сущностей-агрегатов закрыто политикой супер-админа.

## Ограничения и видимость для админов

Две независимые модели (`AdminSubjectAccess` / `AdminGroupAccess`):

- **Restricted (глобальный запрет супер-админа)** — блокирует действия. `AdminAccessService.
  EnsureSubjectAllowedAsync` вызывается в `GradeService` (создание колонок, выставление баллов,
  удаление) и может применяться в других сценариях работы с предметом; при запрете — `403`.
  Управление — супер-админ через `/api/admins/{id}/subjects|groups/.../restriction`.
- **Hidden (локальное скрытие)** — не блокирует, лишь фильтрует собственные списки админа.
  Управление — сам админ через `/api/me/subjects|groups/.../hidden`.

Запись доступа существует, только пока задействован хотя бы один флаг; со снятием обоих — удаляется.

## Защита целостности данных

- Профили `Student`/`Admin` каскадно удаляются вместе с учётной записью (общий PK).
- `Submission`/`StudentGrade` ссылаются на `Admin` с `Restrict`: админа с историей проверок/оценок
  удалить нельзя (`AdminService.DeleteAsync` → `409`). Себя удалить тоже нельзя.
- Регистрация и создание админа выполняются с компенсацией: при сбое создания профиля учётная
  запись откатывается (удаляется).

## Секреты

Ключ JWT, строка подключения, доступы SMTP и стартовые админы **не хранятся в репозитории** —
только в `secrets.json` / user-secrets / переменных окружения. См. [docs/configuration.md](configuration.md).
